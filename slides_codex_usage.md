# Codex 使用指南

**从提问到交付：10 分钟上手**

- 适用对象：开发者、测试、技术团队
- 目标：学会用 Codex 完成常见工程任务

---

## 1. Codex 是什么？

Codex 可以理解为一个“能读代码、改代码、跑命令”的 AI 工程助手：

- 理解仓库结构与上下文
- 编写与修改代码
- 执行测试与静态检查
- 生成提交信息与 PR 描述

> 核心价值：缩短“需求 → 实现 → 验证”的闭环时间。

---

## 2. 典型使用场景

- 新功能开发：根据需求直接产出可运行代码
- Bug 修复：定位报错、修改并补充测试
- 重构优化：提炼函数、统一风格、减少重复逻辑
- 文档与脚本：补 README、写迁移脚本、自动化检查

---

## 3. 推荐工作流（4 步）

1. **明确目标**：一句话描述结果 + 约束
2. **让 Codex 执行**：先小步修改，再逐步扩展
3. **验证结果**：运行测试 / lint / 构建
4. **沉淀产物**：提交 commit + PR 说明

---

## 4. 高质量提问模板

```text
你现在在一个 {技术栈} 项目中。
请完成：{具体任务}
约束：
1) 不改动 {目录/模块}
2) 保持 {风格/兼容性}
3) 补充必要测试
验收标准：
- {可执行命令} 全部通过
- 返回修改摘要与风险说明
```

---

## 5. 示例：新增接口

**需求**：新增 `GET /health` 返回服务状态。

你可以这样说：

- 在现有路由风格下添加 `GET /health`
- 返回 `{ status: "ok", timestamp }`
- 增加单元测试
- 运行测试后汇报结果

---

## 6. 示例：修复 Bug

**需求**：用户分页接口在 `page=0` 时抛错。

提示 Codex：

- 先定位触发条件与调用链
- 最小化修复（避免大规模重构）
- 增加边界测试（`page=0`, `page<0`, `page=1`）
- 输出根因分析

---

## 7. 和 Codex 协作的最佳实践

- **小步快跑**：一次只做一个目标
- **先读后改**：先让 Codex 总结再动手
- **显式约束**：性能、兼容性、安全都要写清楚
- **强制验证**：要求“先跑测试，再汇报”
- **可审计输出**：让它提供修改文件清单与命令结果

---

## 8. 常见问题与规避

- 任务过大导致偏题 → 拆分为多轮任务
- 输出“看起来对”但未验证 → 强制运行命令
- 改动超范围 → 明确“禁止修改目录”
- 风格不一致 → 指定 lint / formatter 规则

---

## 9. 团队落地建议

- 统一提示词模板（需求、约束、验收）
- PR 必须附：改动摘要 + 测试结果 + 风险点
- 引入自动化门禁：CI 失败不可合并
- 先在低风险模块试点，再推广

---

## 10. 一页总结

**Codex = 研发加速器，不是“自动驾驶”**

你需要提供：
- 清晰目标
- 明确约束
- 可执行验收标准

Codex 能提供：
- 更快实现
- 更快验证
- 更高迭代效率
